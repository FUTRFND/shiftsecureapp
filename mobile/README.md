# Mobile build (Capacitor + iOS + Android)

Phase 2 set up everything that doesn't need a Mac or Android Studio. The native
projects themselves (`ios/` and `android/` folders) are generated by Capacitor
**on your local machine** in one of two ways:

1. Clone the repo locally, install deps, and run `npx cap add ios` /
   `npx cap add android` once. Commit the generated folders.
2. Or keep them out of source control and regenerate on each developer
   machine. Either is fine; pick one and stick with it.

The Lovable web preview will keep working unchanged — Capacitor is only used
when packaging for the App Store / Play Store.

---

## One-time setup (local machine)

Prereqs:

- Node 20+, Bun 1.1+
- **iOS:** macOS, Xcode 15+, CocoaPods (`sudo gem install cocoapods`)
- **Android:** Android Studio Hedgehog+, JDK 17, an Android SDK with API 34

```bash
git clone <your repo>
cd <repo>
bun install
cp .env.example .env.local   # fill in Supabase + (later) RevenueCat keys
bun run build:mobile         # produces dist/spa for Capacitor's webDir

npx cap add ios
npx cap add android
npx cap sync
```

After `cap add`, open the native projects:

```bash
npx cap open ios       # launches Xcode with ios/App/App.xcworkspace
npx cap open android   # launches Android Studio
```

## Day-to-day workflow

```bash
# 1. Make changes to src/...
# 2. Rebuild the web bundle
bun run build:mobile

# 3. Sync into the native projects (copies dist/spa + updates plugins)
npx cap sync

# 4. Run on a simulator / device from Xcode or Android Studio,
#    or:
npx cap run ios
npx cap run android
```

`cap sync` runs Pod install on iOS and Gradle resolution on Android. After
adding or removing a Capacitor plugin, always run `cap sync` before launching
the native build.

## Configuration reference

- **App display name:** `Shift Secure`
- **App identifier (bundle id / applicationId):** `com.badexy.shiftsecure`
  (set in `capacitor.config.ts`; this is the value already reserved in App
  Store Connect and the Google Play Console)
- **Web dir:** `dist/spa` (output of `bun run build:mobile`)

### Technical identifiers intentionally not rebranded

The following identifiers still carry the legacy `handoffhero` token. They
are NOT user-facing — they're contract strings the rest of the system is
already wired against. Renaming them would silently break auth callbacks or
RevenueCat product lookups, so they're documented here instead.

| Identifier                         | Where it lives                                              | Why it stays                                                                                       |
| ---------------------------------- | ----------------------------------------------------------- | -------------------------------------------------------------------------------------------------- |
| `handoffhero://` URL scheme        | `VITE_APP_URL_SCHEME`, Info.plist `CFBundleURLSchemes`, Android `<data android:scheme=...>` | Already registered in Supabase Auth → Additional Redirect URLs; changing it invalidates every magic link / password reset email already sent. |
| `handoffhero.app` Universal Link host | `VITE_APP_UNIVERSAL_LINK_HOST`, iOS Associated Domains, Android `<data android:host=...>` | Apple AASA + Google `assetlinks.json` are tied to this domain. Move to a Shift Secure domain only as a coordinated change (new AASA file, new Supabase redirect entries, new entitlement). |
| `handoffhero_pro_monthly` / `handoffhero_pro_annual` product IDs | `src/config/subscription.ts`, App Store Connect, Play Console, RevenueCat dashboard | Product IDs are immutable in App Store Connect / Play Console once created. If the products have not been created yet, you may rename them in all four places at the same time before first publish. |

All three are overridable via env vars (`VITE_APP_URL_SCHEME`,
`VITE_APP_UNIVERSAL_LINK_HOST`) or `src/config/subscription.ts`, so a future
coordinated migration is straightforward — it just isn't a code-only change.
